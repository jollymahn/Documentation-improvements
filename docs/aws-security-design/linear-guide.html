Securing Applications in AWS

Combined Design with Panorama Management — Deployment Guide

1. Overview

This guide deploys Palo Alto Networks VM-Series firewalls in AWS using a combined design with Panorama management. A dedicated Security VPC with Gateway Load Balancer (GWLB) inspects all traffic — outbound, inbound, and east-west — across spoke VPCs connected via AWS Transit Gateway (TGW).

▶ Architecture Summary
ComponentRole
Security VPCHosts VM-Series firewalls behind a GWLB. All spoke VPC traffic is routed here for inspection.
Spoke VPCsApplication workloads (e.g., app01, app02). Connected to TGW.
Management VPCHosts Panorama in active/standby. Connected to TGW.
Transit Gateway (TGW)Hub connecting Security, Spoke, and Management VPCs. Two route tables: security and spokes. Appliance mode enabled on Security VPC attachment.
Gateway Load Balancer (GWLB)Distributes traffic to VM-Series using Geneve encapsulation (UDP 6081). Cross-zone load balancing enabled.
PanoramaCentralized management for VM-Series configuration, licensing, and monitoring. Deployed in Management-Only mode.
Strata Logging Service (SLS)Centralized log collection via Cortex Data Lake.

Key Concept

Each traffic flow type has its own dedicated subinterface and zone on the private dataplane interface (ethernet1/1). This enables granular security policies per traffic direction — inbound, outbound, and east-west are independently controlled. The parent interface (ethernet1/1) remains in the GWLB zone for health probes and untagged traffic. GWLB endpoints are mapped to the appropriate subinterface based on their traffic role.

Overlay Routing Model

In the overlay combined model, additional per-spoke subinterfaces (APP1, APP2, APP3) provide traffic identity at the spoke VPC level. This allows security policies to be written per application rather than just per direction. In the non-overlay model, these per-spoke zones are omitted.

▶ Traffic Flows

Outbound (spoke → internet)

  1. Spoke instance → subnet route table → TGW attachment
  2. TGW spokes route table → Security VPC attachment (static 0.0.0.0/0)
  3. Security VPC TGW subnet route table → GWLB endpoint (same AZ)
  4. GWLB endpoint → GWLB → VM-Series firewall (chosen from target group)
  5. Firewall: receives on ethernet1/1 → associates with OUTBOUND zone (subinterface ethernet1/1.20) → applies security policy → NAT (SNAT to ethernet1/2 IP) → forwards out ethernet1/2
  6. Public subnet route table → IGW → internet

Inbound (internet → spoke, Combined option with GWLB)

  1. Internet → App VPC IGW → IGW edge-associated route table → GWLB endpoint (in app VPC)
  2. GWLB endpoint → GWLB (in Security VPC via PrivateLink) → VM-Series firewall
  3. Firewall: receives on ethernet1/1 → associates with INBOUND zone (subinterface ethernet1/1.10) → applies security policy → returns via same GWLB endpoint
  4. GWLB endpoint → ALB → application instances

East-West (spoke ↔ spoke)

  • Same path as outbound through TGW → GWLB → firewall
  • Traffic associates with EASTWEST zone (subinterface ethernet1/1.30)
  • Dedicated zone enables explicit east-west security policies

Naming Convention

Overlay Routing vs Non-Overlay Routing

The network template (position 4) differs based on your chosen routing model. This is the most important architectural decision in the deployment — it affects how outbound internet traffic egresses the environment and how much operational control you have over NAT.

Overlay routing uses the firewall’s public interface (ethernet1/2) for outbound internet traffic. The firewall performs source NAT on ethernet1/2, and the associated EIP provides the public IP. This gives you full control over NAT, egress IPs, and outbound security policy — all traffic is inspected and NATed by the firewall.

Non-overlay routing uses AWS NAT Gateways for outbound internet traffic. Traffic is inspected by the firewall but returns to the VPC via GWLB, and the NAT Gateway handles the actual internet egress. This simplifies the firewall configuration (no public interface routing needed) but adds NAT Gateway cost and removes direct control over egress IPs.

Important

Even in a non-overlay deployment, it is strongly recommended to keep ethernet1/2 and the public subnet provisioned on each firewall. If the decision is made later to migrate to overlay routing, having the interface and subnet already in place avoids redeploying the firewalls and restructuring the VPC. What are Reference Templates?

A Reference Template is a template that is associated with a device group rather than a template stack. It does not push configuration to firewalls — instead, it makes the template's network objects (zones, interfaces, virtual routers) available for use when building policies in that device group. Without a reference template, the policy editor cannot see or autocomplete zone names, interface names, or other network-layer objects defined in your templates.

Where to apply: Assign the reference template to the child device group where you write security and NAT policies (e.g., DG-AWS-USW2), not the parent. Always use the template stack (e.g., STK-AWS-PRD-USW2-AZ-A) as the reference template rather than an individual template — the stack encompasses all layers in the hierarchy and ensures the policy editor has visibility into every zone, interface, and network object regardless of which template defines it.

Why the name default matters

The name default (all lowercase, case-sensitive) is a functional keyword in Panorama — not just a friendly label. When a Log Forwarding Profile named exactly default exists in a device group, Panorama automatically assigns it to every new security rule created in that device group or its children. This eliminates the most common logging gap: an admin forgetting to attach a profile to a rule, leaving traffic invisible to the SOC.

FeatureProfile named Internal-SIEMProfile named default
Manual selectionRequired for every new ruleNot required (auto-populated)
Risk of missing logsHigh (human error)Low (automatic coverage)
Best use caseRules requiring unique destinationsStandard corporate logging policy

Important considerations:

  • Case-sensitive — the profile must be named exactly default in lowercase
  • Manual override — if you manually select a different profile on a rule, Panorama honors your selection; default only fills the empty slot on new rules
  • Not retroactive — renaming a profile to default does not attach it to existing rules created without a profile; it only applies to new rule creation
  • Placement — create the default profile at the highest applicable parent device group (e.g., DG-AWS) so all child device groups inherit automatic coverage
Pro Tip: Clone the IoT Logging Profile

If you have an IoT subscription, Panorama provides a predefined logging profile with enhanced IoT fields. You can clone this IoT profile and rename the clone to default — this gives you the automatic assignment behavior while retaining the enriched IoT logging fields across your entire policy set.

3.5.4 Override Default Intra-Zone Rule

Because east-west traffic is intra-zone (EASTWEST → EASTWEST), the default intra-zone Allow rule must be overridden:

  1. Navigate to Policies > Security > Default Rules
  2. Select intrazone-default > Click Override
  3. Set Action to Deny
Critical — Why This Matters in Cloud Architectures

By default, PAN-OS allows all intra-zone traffic without requiring an explicit security policy. In traditional on-premises deployments this is often acceptable, but in cloud architectures like GWLB-based designs all traffic is intra-zone — east-west traffic between spoke VPCs enters and exits through the same EASTWEST zone on the same subinterface. Without this override, the default intra-zone Allow rule silently passes all east-west traffic with no policy evaluation, no logging, and no application visibility.

Overriding the intra-zone default to Deny forces all traffic — even between sources and destinations in the same zone — to match an explicit security policy. This means:

  • Every traffic flow must have a rule that permits it (zero-trust enforcement)
  • All permitted traffic is logged and inspected by the threat prevention engine
  • No traffic passes unnoticed through the implicit allow

This is not optional in cloud deployments. Without this override, your east-west security policies in Section 3.5.3 would never be evaluated — the default intra-zone Allow rule matches first and permits everything.

3.7.2 Dynamic Address Groups

▶ 3.8 Bootstrap Configuration

The VM-Series firewalls in this deployment use user-data bootstrap to automatically register with Panorama, receive their configuration, and enable GWLB integration on first boot. This is the recommended bootstrap method for Panorama-managed deployments.

For a comprehensive guide to all VM-Series bootstrap methods, prerequisites, and troubleshooting, refer to the VM-Series Bootstrap Guide — AWS. That guide covers four bootstrap methods ranked by preference:

  1. Panorama Licensing Plugin — auto-scaling, production deployments
  2. Simple Bootstrap (user-data) — used in this deployment
  3. Storage Basic Bootstrap — init-cfg + authcode via S3
  4. Storage Full Bootstrap — full bootstrap.xml via S3

3.8.1 Generate VM Auth Key

The VM auth key allows VM-Series firewalls to authenticate and register with Panorama during bootstrap. Generate this key on the Panorama CLI:

request bootstrap vm-auth-key generate lifetime 8760

Example output:

VM auth key 123456789012345 generated. Expires at: 2025/04/09 14:15:56
  • Lifetime range: 1–8760 hours (1 year max)
  • Save this key — needed for the VM-Series user-data in Section 4.5
